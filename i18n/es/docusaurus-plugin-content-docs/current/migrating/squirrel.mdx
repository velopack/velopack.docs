# Desde Squirrel
<AppliesTo win />
Migra desde `Squirrel.Windows` o `Clowd.Squirrel` a Velopack.

Estos son los pasos generales necesarios:

1. Reemplaza el paquete nuget `Squirrel.Windows` o `Clowd.Squirrel` con el último [Paquete NuGet de Velopack](https://www.nuget.org/packages/velopack).

0. Instala la herramienta de línea de comandos `vpk`, ya que es lo que usarás para construir versiones de Velopack.
   ```cmd
   dotnet tool install -g vpk
   ```

   :::tip
   Asegúrate de estar usando la misma versión de vpk y del paquete de Velopack, esta es la única configuración oficialmente compatible.
   :::

   Alternativamente, puedes ejecutar `vpk` sin instalarlo globalmente usando el comando [`dnx`](https://learn.microsoft.com/en-us/dotnet/core/tools/dotnet-tool-exec?WT.mc_id=DT-MVP-5003472):
   ```cmd
   dnx vpk --version 1.0.0
   ```
   
   Reemplaza `1.0.0` con la versión del paquete de Velopack que estés usando en tu aplicación.

0. Tendrás que reemplazar `Squirrel.SquirrelAwareApp` al principio de tu aplicación por `Velopack.VelopackApp.Build().Run()`.

   Los accesos directos [[Leer más]](integrating/shortcuts.mdx) y las entradas del registro se gestionan automáticamente por ti en Velopack, así que si actualmente estás haciendo esto en los hooks de `SquirrelAwareApp`, deberían eliminarse. Por ejemplo, si tus hooks eran así antes:
   ```cs
   public static void Main(string[] args)
   {
       SquirrelAwareApp.HandleEvents(
           onInitialInstall: OnAppInstall,
           onAppUninstall: OnAppUninstall,
           onEveryRun: OnAppRun);
   }
   
   private static void OnAppInstall(SemanticVersion version, IAppTools tools)
   {
       tools.CreateShortcutForThisExe(ShortcutLocation.StartMenu | ShortcutLocation.Desktop);
   }
   
   private static void OnAppUninstall(SemanticVersion version, IAppTools tools)
   {
       tools.RemoveShortcutForThisExe(ShortcutLocation.StartMenu | ShortcutLocation.Desktop);
   }
   
   private static void OnAppRun(SemanticVersion version, IAppTools tools, bool firstRun)
   {
       if (firstRun) MessageBox.Show("Thanks for installing my application!");
   }
   ```
   Entonces migrarías al siguiente código, eliminando los hooks de accesos directos:
   ```cs
   public static void Main(string[] args)
   {
       // Thank the user for installing the app on first run.
       // Note that the MessageBox class below comes from WinForms or WPF.
       VelopackApp.Build()
           .OnFirstRun(v => MessageBox.Show("Thanks for installing my application!"))
           .Run();
   }
   ```

0. Los hooks de Velopack ya no admiten la creación manual de accesos directos. En su lugar, estos se crean/eliminan automáticamente por ti como parte del instalador.

0. Velopack requiere que la aplicación se reinicie para aplicar las actualizaciones. Las API anteriores aplicaban las actualizaciones descargadas *antes* de desencadenar el reinicio.

0. El concepto de `SquirrelAwareApp` ya no existe, así que si has añadido algún atributo, entradas del manifiesto del ensamblado u otros archivos para indicar que tu binario ahora es compatible, puedes eliminarlos. Cada paquete de Velopack tiene exactamente un binario "VelopackApp", que debe implementar el código anterior en la parte superior de `Main`. De forma predeterminada, Velopack buscará un binario en `{packDir}\{packId}.exe`. Si tu exe principal de VelopackApp tiene un nombre diferente, deberías proporcionar el nombre con el argumento `--mainExe yourApp.exe`.

0. El archivo "RELEASES" ya no es un formato que Velopack utilice, pero producirá uno al construir paquetes en windows con el canal predeterminado (por ejemplo, sin proporcionar ningún argumento de canal). En su lugar, Velopack producirá archivos `releases.{channel}.json`, que deben tratarse de la misma manera. Si deseas migrar una aplicación heredada de windows a Velopack, deberías publicar una versión (omitiendo el argumento `--channel`) y subir tanto el archivo `RELEASES` como el archivo `releases.win.json` que produce Velopack a tu feed de actualizaciones.

0. En general, la línea de comandos admite todas las mismas funciones, pero los nombres de los argumentos o los comandos pueden haber cambiado. Velopack ya no admite tomar un `.nupkg` que fue creado por dotnet o nuget.exe. Deberías compilar/publicar tu aplicación y proporcionar la salida de tu compilador a `vpk pack` en su lugar. Un ejemplo muy simple podría verse así
   ```cmd
   dotnet publish --self-contained -r win-x64 -o publish
   vpk pack -u YourAppId -v 1.0.0 -p publish -e yourMainBinary.exe
   ```

   Por favor, revisa la ayuda de la línea de comandos de vpk para más detalles:
   ```cmd
   vpk -h
   ```
0. En tu aplicación instalada, ya no hay carpetas `app-1.0.0`, y en una aplicación recién instalada no habrá un `YourApp.exe` en la raíz de la aplicación. Cambiar la ruta de la aplicación (por ejemplo, app-1.0.0 y app-1.0.1) como hace Squirrel es muy malo, rompe muchas cosas en Windows. Reglas del firewall, problemas de antivirus, preferencias de GPU, fijación de iconos en la bandeja y más. Mantener la ubicación del exe en la misma posición después de cada actualización soluciona muchos problemas, por lo que velopack siempre almacenará tu programa en `{root}\current\YourApp.exe`. La primera vez que tu aplicación de Squirrel descargue una actualización de Velopack, el actualizador de Velopack realizará una migración de la aplicación, buscará accesos directos en todas las ubicaciones habituales/predeterminadas y los limpiará/reparará para que apunten a la nueva ubicación. Si anteriormente tenías asociaciones de archivos o accesos directos fuera de las ubicaciones predeterminadas que apuntaban a `{root}\YourApp.exe`, estos tendrán que actualizarse, ya que ese archivo ya no existirá en las nuevas instalaciones.

## Diferencias de API con Clowd.Squirrel
Esta lista no es exhaustiva y puede cambiar con el tiempo. Si tienes una pregunta específica, por favor pregunta en el [Discord de Velopack](https://discord.gg/velopack).
Para muchas de estas, un simple buscar/reemplazar en tu base de código debería ser suficiente, sin embargo, estas son solo pautas y deberías revisar el 

0. La clase `ReleaseEntry` se reemplaza por `VelopackAsset`
0. `UpdateManager` ya no es desechable. Elimina cualquier declaración `using`.
0. La propiedad `UpdateManager.IsInstalledApp` se renombra a `UpdateManager.IsInstalled`
0. El método `UpdateManager.CheckForUpdate()` se renombra a `UpdateManager.CheckForUpdatesAsync()`
0. `UpdateInfo.CurrentlyInstalledVersion` se reemplaza comprobando el `UpdateInfo.BaseRelease`
0. `UpdateInfo.ReleasesToApply` ya no existe, dependiendo de tu uso esto se reemplaza por `UpdateInfo.TargetFullRelease` o `UpdateInfo.DeltasToTarget`
0. `UpdateInfo.FutureReleaseEntry` se reemplaza por `UpdateInfo.TargetFullRelease`
0. `UpdateInfo.Create()` ya no existe, usa el constructor directamente `new UpdateInfo()`
0. `UpdateManager.DownloadReleases()` se reemplaza por `UpdateManager.DownloadUpdatesAsync()`
