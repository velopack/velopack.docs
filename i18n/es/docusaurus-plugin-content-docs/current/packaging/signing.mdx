# Firma de código
<AppliesTo win mac />

La firma de código es una parte esencial de la distribución de aplicaciones. En Windows, es probable que las aplicaciones sin firmas de código se marquen como virus. En OSX, la firma de código y la notarización son obligatorias antes de que los usuarios puedan ejecutar tu aplicación.

En ambas plataformas, la firma debe realizarla el propio Velopack, esto se debe a que los binarios de Velopack (como Update y Setup) deben firmarse en diferentes puntos del proceso de compilación del paquete.

## Firma en Windows

Aquí tienes la regla general para las advertencias de Windows Defender/Smart Screen:
- No firmas tu código. Lanzas MyApp-v1.exe. La gente recibe advertencias de smart screen durante un tiempo hasta que aumenta la reputación de ese archivo. Lanzas MyApp-v2.exe. La gente recibe advertencias de nuevo hasta que MyApp-v2.exe acumula suficiente reputación.
- Firmas tu código con un certificado OV. Lanzas MyApp-v1.exe. La gente recibe advertencias de smart screen durante un tiempo hasta que aumenta la reputación de ese certificado. Lanzas MyApp-v2.exe. La gente no recibe advertencias porque el certificado ya tiene una buena reputación. Con el tiempo, tu certificado caduca y llega el momento de lanzar MyApp-v47.exe. Renuevas tu certificado, y la gente recibe advertencias de nuevo hasta que el nuevo certificado acumula reputación.
- Firmas tu código con un certificado EV. La gente nunca recibe advertencias.

### Adquirir un certificado de firma de código

#### Azure Artifact Signing

A partir de abril de 2024, la forma más eficaz de firmar con código tu producto es mediante [**Azure Artifact Signing**](https://azure.microsoft.com/en-us/products/artifact-signing) (anteriormente conocido como Azure Trusted Signing).

- Cuesta 10 USD/mes (o 120 USD/año), lo cual está a la par o es más barato que los certificados de firma de código de otros emisores.
- Solución totalmente en la nube / No requiere que se te envíe ningún hardware físico (HSM).
- La firma se puede automatizar (p. ej. en tu canalización de CI).
- Los certificados de Artifact Signing obtienen reputación instantánea integrada para smart-screen y defender.

Artifact Signing se puede usar siguiendo la [documentación de Microsoft para usar Artifact Signing con signtool.exe](https://learn.microsoft.com/en-us/azure/trusted-signing/how-to-signing-integrations#download-and-install-net-80-runtime), y leyendo luego nuestra documentación a continuación sobre cómo configurar signtool.exe con Velopack.

#### Otros proveedores de certificados

Si en cambio deseas comprar un certificado de firma de código de un proveedor diferente, Microsoft enumera los siguientes emisores de confianza [en su documentación](https://learn.microsoft.com/windows-hardware/drivers/dashboard/code-signing-cert-manage):

- [Certum](https://shop.certum.eu/data-safety/code-signing-certificates/certum-ev-code-sigining.html)
- [SSL.com](https://www.ssl.com/certificates/ev-code-signing/)
- [DigiCert](https://www.digicert.com/signing/code-signing-certificates)
- [Entrust](https://www.entrustdatacard.com/products/digital-signing-certificates/code-signing-certificates)
- [GlobalSign](https://go.microsoft.com/fwlink/p/?LinkId=620888)
- [IdenTrust](https://www.identrust.com/digital-certificates/trustid-ev-code-signing)
- [Sectigo (anteriormente Comodo)](https://sectigo.com/ssl-certificates-tls/code-signing)

También hay revendedores que pueden tener certificados de emisores autorizados a precios más bajos, pero el riesgo añadido es que no obtendrás soporte oficial para esos certificados y los procedimientos de verificación de identidad pueden ser más enrevesados.
    
:::warning
Ten en cuenta que desde el 1 de junio de 2023 [ha habido un cambio de política](https://knowledge.digicert.com/general-information/new-private-key-storage-requirement-for-standard-code-signing-certificates-november-2022), lo que significa que ya no es posible descargar tu certificado de firma de código. Debe almacenarse en un HSM en la nube autorizado o se te enviará un HSM USB. Muchas de estas soluciones requerirán pasos manuales y no serán compatibles con la automatización / CI. Revisa cuidadosamente antes de comprar.
:::

### Firma mediante Azure Artifact Signing {#signing-via-azure-artifact-signing}

[Azure Artifact Signing](https://learn.microsoft.com/azure/trusted-signing/overview) (anteriormente Trusted Signing) es un servicio de firma gestionado de Microsoft y una de las formas más [rentables](https://azure.microsoft.com/en-us/pricing/details/artifact-signing) de firmar tu aplicación.

Para obtener información detallada sobre Azure Artifact Signing, consulta la [documentación oficial](https://learn.microsoft.com/azure/trusted-signing/).

:::note
La firma depende de `signtool.exe`, que solo es compatible con Windows. Si estás usando un sistema operativo diferente, tendrás que firmar tus binarios en una máquina Windows antes de desplegarlos.
:::

1. Primero tendrás que crear una cuenta de Azure en: https://azure.microsoft.com/pricing/purchase-options/azure-account. Esta cuenta deberá tener una [suscripción activa](https://learn.microsoft.com/azure/cost-management-billing/manage/create-subscription#create-a-subscription).
0. Registra el [proveedor de recursos](https://learn.microsoft.com/azure/azure-resource-manager/management/resource-providers-and-types) de Azure Artifact Signing. Esto puede hacerse en el portal de Azure, o mediante la CLI de Azure siguiendo [estas instrucciones](https://learn.microsoft.com/azure/trusted-signing/quickstart?tabs=registerrp-portal%2Caccount-portal%2Corgvalidation%2Ccertificateprofile-portal%2Cdeleteresources-portal#register-the-trusted-signing-resource-provider).
0. Crea el recurso de Azure Artifact Signing siguiendo [estas instrucciones](https://learn.microsoft.com/azure/trusted-signing/quickstart?tabs=registerrp-portal%2Caccount-portal%2Corgvalidation%2Ccertificateprofile-portal%2Cdeleteresources-portal#create-a-trusted-signing-account). Anota el nombre y la región que selecciones, ya que usarás estos valores más adelante para los valores `CodeSigningAccountName` y `Endpoint` respectivamente, en el archivo JSON de configuración de firma.
0. Crea una solicitud de validación de identidad siguiendo [estas instrucciones](https://learn.microsoft.com/azure/trusted-signing/quickstart?tabs=registerrp-portal%2Caccount-portal%2Corgvalidation%2Ccertificateprofile-portal%2Cdeleteresources-portal#create-an-identity-validation-request). Este paso debe realizarse en el Portal de Azure. La validación de identidad puede hacerse para una organización o para un desarrollador individual. Este proceso puede tardar un tiempo, ya que Microsoft verifica la información de identidad. Vuelve a estos pasos una vez completado el proceso de validación de identidad.
0. Crea un perfil de certificado siguiendo [estas instrucciones](https://learn.microsoft.com/azure/trusted-signing/quickstart?tabs=registerrp-portal%2Caccount-portal%2Corgvalidation%2Ccertificateprofile-portal%2Cdeleteresources-portal#create-a-certificate-profile). Hay múltiples [modelos de confianza](https://learn.microsoft.com/azure/trusted-signing/concept-trusted-signing-trust-models) compatibles. El modelo de confianza pública (Public Trust) es el más común y es adecuado para la mayoría de las aplicaciones. Anota el nombre del perfil de certificado, ya que usarás este valor más adelante como `CertificateProfileName` en el archivo JSON de configuración de firma.
0. Velopack viene incluido con una versión de [signtool.exe](https://learn.microsoft.com/azure/trusted-signing/how-to-signing-integrations#download-and-install-signtool) (x64) y el [paquete dlib](https://learn.microsoft.com/azure/trusted-signing/how-to-signing-integrations#download-and-install-the-trusted-signing-dlib-package) que es compatible con Azure Artifact Signing. Estos componentes [requieren que el .NET 8.0 Runtime](https://learn.microsoft.com/azure/trusted-signing/how-to-signing-integrations#download-and-install-net-80-runtime) esté instalado en la máquina que realizará la firma. Puedes encontrar la última versión del .NET 8.0 runtime [aquí](https://dotnet.microsoft.com/download/dotnet/8.0).
0. Configura la autenticación con Azure Artifact Signing. `signtool.exe` aprovecha la clase común [DefaultAzureCredentials](https://learn.microsoft.com/dotnet/api/azure.identity.defaultazurecredential?view=azure-dotnet) para autenticarse con Azure. Esta clase admite la autenticación de varias formas, incluida la CLI de Azure. Después de instalar la CLI de Azure, ejecuta `az login`. Esto proporcionará a signtool.exe la autenticación que necesita cuando `vpk` lo invoque. Si te estás ejecutando dentro de [GitHub Actions](https://github.com/marketplace/actions/azure-cli-action) o [Azure DevOps](https://learn.microsoft.com/azure/devops/pipelines/tasks/reference/azure-cli-v2?view=azure-pipelines), hay soporte integrado que gestiona la autenticación necesaria.
0. Crea el archivo JSON de metadatos de firma. Este archivo tiene tres parámetros requeridos y un CorrelationId opcional. El archivo debería verse así:
   ```json
   {
     "Endpoint": "<Artifact Signing account endpoint>",
     "CodeSigningAccountName": "<Artifact Signing account name>",
     "CertificateProfileName": "<Certificate profile name>"
   }
   ```
   Los valores de cada uno de estos se pueden encontrar en los pasos anteriores donde creaste el recurso de Azure Artifact Signing y el perfil de certificado. El archivo debe guardarse con codificación UTF-8 (sin BOM). Aunque no es obligatorio, a menudo se nombra como `metadata.json`. Los detalles completos se pueden ver [aquí](https://learn.microsoft.com/azure/trusted-signing/how-to-signing-integrations#create-a-json-file).
0. Pasa el archivo JSON de metadatos. Al llamar a `vpk pack` usando `--azureTrustedSignFile <path to json file>`. `vpk` pasará el archivo a `signtool.exe` para firmar los recursos necesarios.

### Firma mediante `signtool.exe`
Normalmente la firma se realiza mediante `signtool.exe`. Si ya usas esta herramienta para firmar tu aplicación, puedes pasar tus parámetros de firma directamente a Velopack (menos el comando 'sign').

Por ejemplo, si tu comando de firma anterior era:
```cmd
signtool.exe sign /td sha256 /fd sha256 /f yourCert.pfx /tr http://timestamp.comodoca.com
```

Entonces ahora con `--signParams` sería:
```cmd
vpk pack ... --signParams "/td sha256 /fd sha256 /f yourCert.pfx /tr http://timestamp.comodoca.com"
```

:::tip
Siempre deberías pasar rutas absolutas en tus argumentos de firma, ya que no hay garantía de que signtool se ejecute en el mismo directorio de trabajo que vpk.
:::

Si eres nuevo en el uso de `signtool.exe`, puedes consultar la [referencia de línea de comandos aquí](https://learn.microsoft.com/dotnet/framework/tools/signtool-exe). Recomiendo conseguir que la firma funcione primero en un solo binario, usando `signtool.exe`, antes de intentar que las cosas funcionen con la CLI de Velopack.

Si estás usando un USB / HSM y tienes el siguiente error de firma `Error: SignerSign() failed." (-2147023673/0x800704c7)`, esto significa que falta una contraseña / token de desbloqueo. Esto se puede añadir a tu comando de firma con los parámetros `/csp /k`. [[Lee más]](https://stackoverflow.com/questions/17927895/automate-extended-validation-ev-code-signing-with-safenet-etoken/54439759#54439759)

:::warning
Ten cuidado al proporcionar parámetros con espacios. Debes envolver entre comillas cualquier cosa que contenga espacios y escaparlas con una barra invertida. Por ejemplo, `--signParams "/n \"My Cert Name\""`. Si estás usando una tarea `<Exec>` de MSBuild o similar, tendrás que usar `&quot;` en su lugar.
:::

De forma predeterminada, Velopack firmará 10 archivos por llamada a `signtool.exe`, para acelerar la firma y reducir el número de veces que necesitas interactuar con la consola si estás usando algún método de firma interactivo. Esto se puede desactivar con el argumento `--signParallel 1`.

### Comandos y herramientas de firma personalizados
Si tienes requisitos de firma más avanzados, como una herramienta de firma personalizada (p. ej. `AzureSignTool.exe`), entonces puedes proporcionar una plantilla de comando en su lugar, donde `{{file}}` es el binario que Velopack sustituirá y firmará:

```cmd
vpk pack ... --signTemplate "AzureSignTool.exe sign ... {{file}}"
```
Lo anterior firmará un archivo a la vez, pero si tu herramienta de firma admite la firma por lotes de varios archivos a la vez, puedes especificar `{{file...}}` en su lugar.

## Firma multiplataforma
[JSign](https://ebourg.github.io/jsign/) es una versátil herramienta de firma de código que se ejecuta en Linux, macOS y Windows. Admite una amplia gama de algoritmos de firma y servicios de marca de tiempo.

Para usarla con Velopack, haz lo siguiente:
1. Sigue los pasos anteriores para configurar [Azure Artifact Signing](#signing-via-azure-artifact-signing)
0. Descarga la última versión de JSign desde https://ebourg.github.io/jsign/#files
0. Instala la [CLI de Azure](https://learn.microsoft.com/cli/azure/) e inicia sesión con `az login`
0. Obtén un [token de acceso](https://learn.microsoft.com/cli/azure/account?view=azure-cli-latest#az-account-get-access-token) ejecutando `az account get-access-token --resource https://codesigning.azure.net`
0. Usa la opción `--signTemplate` para firmar tus archivos.

```sh
vpk pack ... --signTemplate "jsign --storetype TRUSTEDSIGNING --keystore <region> --storepass <api-access-token> --alias <account>/<profile> {{file}}"
```

## Firma y notarización en OSX
La firma de código y la notarización son obligatorias antes de que los usuarios puedan ejecutar tu aplicación, por lo tanto es un paso necesario antes de desplegar tu aplicación.

### Crear certificados de firma de código
1. Primero, tendrás que crear una cuenta en https://developer.apple.com, pagar la cuota anual de desarrollador y aceptar cualquier acuerdo de licencia. 
0. Navega a tus certificados: https://developer.apple.com/account/resources/certificates
0. Haz clic en el icono (+) para crear nuevos certificados. Necesitas crear tanto un certificado `Developer ID Installer` como uno `Developer ID Application` para la distribución de apps Velopack fuera de la Mac App Store. 
   ![lista de certificados de apple](/images/apple_certificate_list.png)
0. Abre ambos certificados haciendo clic en ellos, pulsa Download, y luego haz doble clic en el archivo ".cer" para instalarlo en tu llavero local.

:::warning
Si tus certificados aparecen como no confiables en Acceso a Llaveros, o recibes una advertencia como **"Warning: unable to build chain to self-signed root for signer "Developer ID Application: YourName"**,
probablemente te falte un certificado de desarrollo en tu máquina que puede descargarse desde aquí: https://www.apple.com/certificateauthority/
:::

### Configurar un perfil de NotaryTool
1. Crea una contraseña específica de la app: https://support.apple.com/en-us/102654. Esta contraseña solo se te mostrará una vez, así que guárdala o anótala en algún lugar.
0. Encuentra tu ID de equipo de apple: https://developer.apple.com/account#MembershipDetailsCard
0. Almacena las credenciales de tu cuenta de Apple en un nuevo perfil de NotaryTool:
   ```sh
   xcrun notarytool store-credentials \
       --apple-id "yourapple@account.com" \
       --team-id "your-located-team-id" \
       --password "your-generated-app-specific-password" \
       "your-local-profile-name-here"
   ```

### Juntándolo todo
Ahora que tienes tu perfil de NotaryTool y los certificados de firma de código instalados, puedes añadir los siguientes parámetros a tu comando `pack`:

```sh
vpk pack \
    ... 
    --signAppIdentity "Developer ID Application: Your Name" \
    --signInstallIdentity "Developer ID Installer: Your Name" \
    --notaryProfile "your-local-profile-name-here" \
```

Cuando estos parámetros se especifican y son válidos, Velopack firmará con código y notarizará automáticamente tu aplicación y los paquetes del instalador.

:::tip
Velopack usará `codesign` y `xcrun notarytool` para firmar y notarizar tu aplicación. De forma predeterminada usarán el llavero predeterminado de tu dispositivo para descubrir certificados y el perfil de notarización.
Si deseas usar un llavero diferente, debes instalar tus certificados y crear tu perfil de notarización en ese llavero y usar el argumento `--keychain {path}` de Velopack
para especificar que Velopack debe descubrir los certificados desde un llavero no predeterminado.
<hr />
Si no proporcionas un archivo de entitlements para hardened runtime, se te proporcionará automáticamente uno adecuado para la mayoría de las apps dotnet. Si tu app requiere
entitlements especiales y los predeterminados no son adecuados, deberías suministrar tus propios entitlements de firma usando `--signEntitlements {path}`
<hr />
Aunque tu certificado pueda aparecer como `Developer ID Application: Your Name (TeamName)` en Acceso a Llaveros, no deberías especificar el nombre del equipo como parte de
tus argumentos `--signAppIdentity` o `--signInstallIdentity`. Tu certificado debería ser simplemente `Developer ID Application: Your Name` sin equipo.
:::

### Automatizar la firma en CI/CD (Github Actions)
También es posible almacenar tus certificados y credenciales de notarización como Action Secrets y firmar tu código durante las compilaciones de CI.

<Tabs groupId="ci" queryString>
<TabItem value="github" label="Github Actions">
1. Inicia Acceso a Llaveros y abre el panel "My Certificates".

0. Selecciona cada certificado (uno a la vez), haz clic derecho y selecciona "Export". Guárdalo como un archivo p12 y anota la contraseña. Puedes usar la misma contraseña para ambos certificados.
   :::tip
   Si no ves la opción de exportar, o exportar como `.p12` está deshabilitado, es posible que necesites cambiar qué llavero o pestaña estás viendo. Deberías estar en la pestaña "My Certificates". 
   Consulta https://stackoverflow.com/questions/15662377/unable-to-export-apple-production-push-ssl-certificate-in-p12-format para obtener más información.
   :::

0. Copia el contenido del certificado al portapapeles como base64, ejemplo:
   ```sh
   base64 -i CERT.p12 | pbcopy
   ```

0. Crea 7 [Github Secrets](https://docs.github.com/actions/security-guides/using-secrets-in-github-actions) para tus flujos de trabajo de Actions
   - `BUILD_CERTIFICATE_BASE64` (b64 de tu certificado de app)
   - `INSTALLER_CERTIFICATE_BASE64` (b64 de tu certificado de instalador)
   - `P12_PASSWORD` (contraseña de los certificados)
   - `APPLE_ID` (tu nombre de usuario de apple)
   - `APPLE_PASSWORD` (tu contraseña específica de la app del paso de notarización anterior)
   - `APPLE_TEAM` (tu id de equipo del paso de notarización anterior)
   - `KEYCHAIN_PASSWORD` (puede ser cualquier cadena aleatoria, se usará para crear un nuevo llavero)

0. Añade un paso a tu flujo de trabajo que instale los certificados y el perfil del llavero. Aquí tienes un ejemplo:
   ```txt
   name: App build & sign
   on: push
   jobs:
     build_with_signing:
       runs-on: macos-latest
       steps:
         - name: Checkout repository
           uses: actions/checkout@v4

         - name: Install Apple certificates and notary profile
           env:
             BUILD_CERTIFICATE_BASE64: ${{ secrets.BUILD_CERTIFICATE_BASE64 }}
             INSTALLER_CERTIFICATE_BASE64: ${{ secrets.INSTALLER_CERTIFICATE_BASE64 }}
             P12_PASSWORD: ${{ secrets.P12_PASSWORD }}
             APPLE_ID: ${{ secrets.APPLE_ID }}
             APPLE_PASSWORD: ${{ secrets.APPLE_PASSWORD }}
             APPLE_TEAM: ${{ secrets.APPLE_TEAM }}
             KEYCHAIN_PASSWORD: ${{ secrets.KEYCHAIN_PASSWORD }}
           run: |
             # create variables for file paths
             CERT_BUILD_PATH=$RUNNER_TEMP/build_certificate.p12
             CERT_INSTALLER_PATH=$RUNNER_TEMP/installer_certificate.p12
             KEYCHAIN_PATH=$RUNNER_TEMP/app-signing.keychain-db
   
             # import certificates from secrets
             echo -n "$BUILD_CERTIFICATE_BASE64" | base64 --decode -o $CERT_BUILD_PATH
             echo -n "$INSTALLER_CERTIFICATE_BASE64" | base64 --decode -o $CERT_INSTALLER_PATH
   
             # create temporary keychain
             security create-keychain -p "$KEYCHAIN_PASSWORD" $KEYCHAIN_PATH
             security set-keychain-settings -lut 21600 $KEYCHAIN_PATH
             security unlock-keychain -p "$KEYCHAIN_PASSWORD" $KEYCHAIN_PATH
   
             # import certificates to keychain
             security import $CERT_BUILD_PATH -P "$P12_PASSWORD" -A -t cert -f pkcs12 -k $KEYCHAIN_PATH
             security import $CERT_INSTALLER_PATH -P "$P12_PASSWORD" -A -t cert -f pkcs12 -k $KEYCHAIN_PATH
             security list-keychain -d user -s $KEYCHAIN_PATH
   
             # create notarytool profile
             xcrun notarytool store-credentials --apple-id "$APPLE_ID" --team-id "$APPLE_TEAM" --password "$APPLE_PASSWORD" velopack-profile

         - name: Compile your app
           ...

         - name: Create Velopack Release
           run: |
             dotnet tool install -g vpk
             vpk pack ... \
             --signAppIdentity "Developer ID Application: Your Name" \
             --signInstallIdentity "Developer ID Installer: Your Name" \
             --notaryProfile "velopack-profile" \
             --keychain $RUNNER_TEMP/app-signing.keychain-db

         # Or use dnx instead of installing globally:
         # - name: Create Velopack Release
         #   run: |
         #     dnx vpk --version 1.0.0 pack ... \
         #     --signAppIdentity "Developer ID Application: Your Name" \
         #     --signInstallIdentity "Developer ID Installer: Your Name" \
         #     --notaryProfile "velopack-profile" \
         #     --keychain $RUNNER_TEMP/app-signing.keychain-db
         # Replace 1.0.0 with your Velopack package version
         # Learn more: https://learn.microsoft.com/dotnet/core/tools/dotnet-tool-exec?WT.mc_id=DT-MVP-5003472

         - name: Clean up keychain
           if: ${{ always() }}
           run: security delete-keychain $RUNNER_TEMP/app-signing.keychain-db
   ```

          security import $CERT_INSTALLER_PATH -P "$P12_PASSWORD" -A -t cert -f pkcs12 -k $KEYCHAIN_PATH
          security list-keychain -d user -s $KEYCHAIN_PATH

          # create notarytool profile
          xcrun notarytool store-credentials --apple-id "$APPLE_ID" --team-id "$APPLE_TEAM" --password "$APPLE_PASSWORD" --keychain $KEYCHAIN_PATH velopack-profile
        env:
          BUILD_CERTIFICATE_BASE64: $(BUILD-CERTIFICATE-BASE64)
          INSTALLER_CERTIFICATE_BASE64: $(INSTALLER-CERTIFICATE-BASE64)
          P12_PASSWORD: $(P12-PASSWORD)
          APPLE_ID: $(APPLE-ID)
          APPLE_PASSWORD: $(APPLE-PASSWORD)
          APPLE_TEAM: $(APPLE-TEAM)
          KEYCHAIN_PASSWORD: $(KEYCHAIN-PASSWORD)
        displayName: 🛠️ Install Apple certs and notary profile
